# 微信公众号正文 · Markdown 生成规范

供 **LLM 系统提示词** 或 **人工校对** 使用。插件「扩写」时本文件与 `公众号扩写规则.md` **合并**为一条 system 消息。渲染与 sanitize 见源码 `src/wechatHtml.ts`；版式示例见源码仓库 `fixtures/sample-wechat.md`。

## 1. 标题

- 正文顶部的 **`# 标题`** 会与草稿箱「标题」字段去重；避免在正文里再重复同一句话。
- 篇内大节用 **`#`** 或 **`##`** 均可，保持层级稳定（不要跳级混乱）。
- 小节若用 **`## 1. xxx`** 这种编号标题，与正文中的 **`1.` 有序列表**不要混在同一视觉块里造成歧义（当前示例用「## 编号」作小标题，列表另起一段）。

## 2. 列表（最重要）

### 有序列表 `1. 2. 3.`

- **优先写「紧凑列表」**：相邻两条之间**不要空行**，减少解析成「松散列表」、进而在编辑器里出现多余空行或编号错位。
- 若一条里需要多段说明，**续行缩进 4 个空格**（或一个 Tab），使其明确属于上一条。

### 无序列表 `*` / `-`

- 同上，多条之间尽量**不要无故空行**。
- **同一条内的第二段、短句补充**（如「重复三次。」「以上。」）：必须 **缩进 4 空格** 写在上一行下面；不要顶格单独成行（否则会被当成列表外的段落）。
- **同一条内较长续写**（如先写问题，再写「重点是……」）：同样用 **缩进 4 空格** 起行，与上一条 `*` 对齐为同一条 `li`。

### 不要

- 在列表标记与正文之间插入 **零宽字符**（ZWSP 等），会导致「幽灵列表项」。
- 用裸的 `1)`、`（1）` 冒充列表（若需编号请用标准 `1.` Markdown）。

## 3. 图片

- 线上草稿：`![说明文字](https://…)`，使用可公网访问的 **HTTPS**。
- 本地预览：`file:///…` 仅开发机可用；成稿应替换为素材库 URL 或由插件上传替换。

## 4. 强调与引用

- 正常用 `**加粗**`；避免整段只用加粗堆叠。
- 需要引用框时用 Markdown 引用 `>`；与列表嵌套时尽量简化结构。

## 5. 段落与对话

- 对话、叙事用**普通空行分段**即可；避免在列表与后续正文之间多插空行却未缩进（易被解析成「列表已结束」）。

## 6. 自检清单（生成后扫一眼）

1. 列表续行是否**缩进**，而非顶格？
2. 有序列表项之间是否**无多余空行**（除非刻意要松散列表）？
3. 首行 `#` 是否与将要填入的草稿标题重复？
4. 图片链接是否为 **https**（正式稿）？

## 附录：可粘贴进 LLM 系统提示词的短版

将下面整段加入「输出格式」或「写作约束」即可：

> 你输出的是**微信公众号正文**用的 Markdown（非技术文档）。请遵守：
> 1. **列表**：有序列表 `1. 2. 3.` 相邻项之间不要空行；无序列表用 `*`，同一条内的续段、补充句必须**缩进 4 个空格**写在上一行下面，**禁止**把仍属于该条的句子顶格单独成行。
> 2. **标题**：篇内 `#` / `##` 层级清晰；不要把与草稿标题栏相同的句子在正文里再写一遍当大标题。
> 3. **图片**：正式稿使用 `![说明](https://…)`；仅在本地预览时用 `file://`。
> 4. 列表标记与文字之间不要插入零宽字符或异常空格。
